# Zug Gate-Based Context Design

**Date:** 2026-04-24
**Status:** Approved

---

## Problem

The current Zug implementation has PERSONA and PLAYBOOK data in context but no forcing function for *activating* it. The instruction "internalize the result. Just use it." is a rule with an infinite opt-out path. The model reads the fingerprint, proceeds normally, and can satisfy "use it" by having read it. The result: patterns from PERSONA appear occasionally and generically rather than holistically reshaping the session arc and interaction texture.

Two distinct failures:
1. **Arc**: No gates govern *when* Zug shifts mode, saves an observation, pushes harder, or backs off. All arc decisions are rule-based — easy to rationalize past.
2. **Texture**: The full PERSONA is a prose blob. There's no pre-selected activation context — no concrete set of patterns the agent is expected to check against at any given moment.

---

## Design

### Overview

Three changes form a chain:

```
synthesis (end of session)
  └── Haiku generates active patterns block → stored as ~/.zug/ACTIVE.md

session start
  └── zug_get_context returns ACTIVE.md block at top + full PERSONA below
  └── HARD GATE: active patterns identified → behavioral frame set → only then respond

during session
  └── gates govern: when to shift mode, when to save an observation, when to push harder
```

### 1. Active Patterns Block (`~/.zug/ACTIVE.md`)

A short file generated by Haiku at the end of each synthesis run. Contains 3-5 behavioral instructions derived from the full PERSONA and latest session observations. Written as direct agent instructions, not observations.

**Format:**
```markdown
## Active Patterns
- Resists new ideas before accepting — don't back off at first pushback, hold the thread
- Top-down thinker — lead with vision or principle, detail comes after
- Diagnoses before reporting — trust the framing they bring, probe the diagnosis not the symptom
- Gets stuck at abstraction → needs a concrete example to unlock the principle
- High action orientation — introduce friction before they commit to a direction
```

Each pattern must pass two gates before being written:
- Is this directly supported by an observation from PERSONA or this session?
- Is this specific enough to change behavior, or generic advice?

Only if both: include it.

### 2. `zug_get_context` Response Structure

The tool returns the active patterns block at the top, full PERSONA below. If `ACTIVE.md` doesn't exist (first session, or synthesis hasn't run yet), the active patterns section is omitted — same graceful handling as empty PERSONA.

```
# Zug Context
Sessions: N | Observations: N

## Active Patterns
...ACTIVE.md content...

## Cognitive Fingerprint
...full PERSONA...

## Playbook
...PLAYBOOK...
```

The gate at session start references the Active Patterns block specifically — not the full PERSONA. This is what makes the gate concrete and checkable.

### 3. Gate Sequences in `zug.md`

The current rule-based instructions are replaced with four explicit gate sequences.

**Session start gate**
```
HARD GATE: After zug_get_context fires
→ Active Patterns block is in context
→ Identify which 2-3 patterns are most relevant to the user's first message
→ Set behavioral frame: challenge intensity, communication style, what to watch for
→ Only then: respond
```

**Mode gate** (replaces "Three Modes — you read which one")
```
Each message arrives →
→ Does this signal a mode change from current mode?
→ If yes: which patterns from Active Patterns apply to the new mode?
→ Adjust behavioral frame
→ Then: respond
```

**Observation gate** (replaces "Always Collecting")
```
Something notable happens →
→ Does an existing PERSONA pattern explain this, or is this new/contradicting?
→ If new or contradicting AND confidence is medium/high: call zug_save_observation
→ Otherwise: continue without saving
```

**Session end gate**
```
Wind-down detected (shorter responses, topic closing, "thanks", silence) →
→ Is there a summary worth writing?
→ Write one-paragraph summary
→ Call zug_end_session with session_id and summary
→ Done
```

The identity sections (Honest Socratic, The ZUG, session modes) are preserved — they define *who* Zug is. The gates define *when and how* that identity activates.

### 4. Gate-Based Synthesis Instructions

The `## Rules` block in `synthesize.ts` is replaced with explicit gates:

**Before adding anything new to PERSONA:**
```
→ Quote the exact text from this session's observations that supports it
→ Is that a direct observation, or an inference?
→ Only if direct: add it
```

**Before removing or significantly rewording an existing PERSONA line:**
```
→ Quote the exact observation from this session that contradicts it
→ Only if you have an explicit contradiction: remove or reword it
→ If uncertain: leave it and add a dated note below it
```

**Before adding to PLAYBOOK:**
```
→ Is this a universal pattern across sessions, or specific to this session?
→ Only if universal: add it
```

**Before updating PLAYBOOK:**
```
→ Does this session's evidence strengthen, weaken, or nuance the existing entry?
→ Only if it changes the meaning: update it
```

The concrete test: "Do I have a quoted observation?" is answerable. "Do I have evidence?" is not.

**Synthesis also generates the Active Patterns block** as a third output:

```
## Your Third Task
From everything you know about this person — the updated PERSONA, this session's
observations, the session summary — write 3-5 active patterns for the NEXT session.

Before writing each pattern:
→ Is this directly supported by an observation from PERSONA or this session?
→ Is this specific enough to change behavior, or is it generic advice?
→ Only if both: include it

Format: one line per pattern, written as a direct behavioral instruction to an agent.
Write as: "when X → do Y" or "don't Z until W"

<ACTIVE>
(active patterns content)
</ACTIVE>
```

## Files Changed

| File | Change |
|---|---|
| `src/storage.ts` | Add `readActive()` / `writeActive()` for `~/.zug/ACTIVE.md` |

| `~/.claude/rules/zug.md` | Rules → gate sequences (same content as above) |

---

## Data Flow

```
zug_end_session called
  └── session log written
  └── synthesize() runs:
        → gate-based PERSONA update
        → gate-based PLAYBOOK update
        → active patterns block generated → writeActive()

next session start
  └── zug_get_context called
        → readActive() → prepended to response
        → readPersona() → appended below
        → readPlaybook() → appended below
  └── HARD GATE: behavioral frame set from active patterns
  └── session proceeds with gates governing arc
```

---

## Out of Scope

- Changes to observation types or confidence levels
- Changes to `zug_get_recent_sessions` or `zug_status`
- Multi-machine sync (Phase 3)
- Onboarding flow (Phase 4)
